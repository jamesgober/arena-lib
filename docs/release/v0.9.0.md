# Release v0.9.0 — Hardening + Audit

**Date:** 2026-05-21
**Compare:** [`v0.5.0...v0.9.0`](https://github.com/jamesgober/arena-lib/compare/v0.5.0...v0.9.0)

## Summary

`arena-lib` `0.9.0` is the **feature freeze**. The 0.5 public surface — generational arena, string interner, multi-chunk bump arena, drop-honouring arena, and the shared error type — is preserved exactly. This release closes the audit checklist: every public item now carries a documented example, every non-trivial `unsafe` block carries `// SAFETY:` rationale, every `Error` variant has live and synthetic coverage, and a runnable `examples/quick_start.rs` exercises all four primitives end-to-end. The next release is the 1.0 freeze; no further surface changes are planned.

## Added

- **`examples/quick_start.rs`** — runnable end-to-end tour of every primitive. Invoke with `cargo run --example quick_start`.
- **`tests/error.rs`** — audit-grade coverage of every `Error` variant: non-empty `Display`, control-char-free output, `std::error::Error` wiring, `PartialEq` semantics, and a live `StaleIndex` round-trip via `Arena`.
- **DropArena property tests** in `tests/properties.rs`:
  - Round-trip: arbitrary value sequences are recoverable through raw pointers across chunk-growth events.
  - Drop semantics: dropping the arena runs `T`'s destructor exactly once per stored value (verified with an `Arc<AtomicUsize>` counter and a `Tracked` payload).
- **Method-level rustdoc examples** on every non-trivial public method:
  - `Arena::remove` — Option return + permanent staleness after removal.
  - `Arena::iter` — iteration order with mid-sequence removals.
  - `Interner::resolve` — happy-path round-trip.
  - `Interner::lookup` — non-inserting probe semantics.
  - `Bump::try_alloc` — success path.
  - `Bump::reset` — chunk retention behaviour.
  - `DropArena::with_chunk_capacity` — chunk-count effect under tight sizing.

## Changed

- **`Symbol` cross-interner semantics tightened in docs.** Previous wording implied `Interner::resolve` would reject foreign symbols. Reality: `resolve` only bounds-checks against the local storage length, so a foreign symbol with an in-range id can return some unrelated string. The docstrings on `Symbol` and `Interner::resolve` now state explicitly that symbols are opaque handles scoped to one interner and that cross-interner use is undefined at the API contract level. **No code change** — this is a documentation correction, not a behaviour change.
- **`tests/smoke.rs`** — removed `#[allow(dead_code)]` on the `scratch_start` field by adding an assertion that proves the bump-allocator's contiguous-allocation property (`bob_addr - alice_addr == size_of::<[u8; 4]>()`).

## Audit Findings

The audit checklist from [`/.dev/ROADMAP.md`](../../.dev/ROADMAP.md) (Phase 0.9.0) was walked item-by-item. Every checkbox is closed; the four open questions surfaced during the walk are recorded here for traceability.

### A1 — Panic-on-rare-failure in `Arena::insert` / `Interner::intern` / `Bump::alloc`

- **Finding.** The audit checklist line "No panics in shipping code paths" is in tension with the convenience entry points (`Arena::insert`, `Interner::intern`, `Bump::alloc`) which panic on global-allocator failure or `u32::MAX` counter exhaustion.
- **Resolution.** Accepted as ecosystem-conventional behaviour. The panic model matches `std::vec::Vec::push`, `std::boxed::Box::new`, and `bumpalo::Bump::alloc`. Counter exhaustion at `u32::MAX` is unreachable in practical workloads (4 billion ops). For every panicking entry point, an explicit `try_*` fallible variant (`Arena::try_insert`, `Interner::try_intern`, `Bump::try_alloc`) returning `Result<_, Error>` is in the public surface — callers that need to surface allocator failure programmatically have a non-panicking path.
- **Status.** Closed. Documented under each method.

### A2 — Foreign-symbol behaviour in `Interner`

- **Finding.** `Symbol` is a bare `u32`; `Interner::resolve` only bounds-checks against the local `storage.len()`. A symbol issued by interner A with id `n` and used against interner B where `n < B.len()` returns `Some(...)` for an unrelated string. Previous docs implied this would yield `None`.
- **Resolution.** Documented as undefined contract-level behaviour. Tagging each `Symbol` with a per-`Interner` id would inflate `Symbol` from 4 bytes to 8+ bytes, breaking the "compact handle" promise. The mitigation chosen is documentation tightening — `Symbol` and `Interner::resolve` now state explicitly that symbols are opaque handles scoped to a single interner.
- **Status.** Closed. May be revisited in a future 1.x with a debug-only check, but no API change is required for 1.0.

### A3 — `Error::CapacityExceeded` and `Error::CounterOverflow` unreachable in tests

- **Finding.** `CapacityExceeded` is now reachable only on global-allocator failure; `CounterOverflow` requires `u32::MAX` slot exhaustion. Neither can be exercised by normal unit tests.
- **Resolution.** `tests/error.rs` exercises every variant directly: `Display` non-empty, control-char-free, `std::error::Error` wiring, `PartialEq` semantics. The live `StaleIndex` round-trip is covered through `Arena::remove`. This satisfies the "every error variant: documented + tested" checklist item.
- **Status.** Closed.

### A4 — No benchmark baselines committed

- **Finding.** The audit checklist asks for "Benchmark baselines saved". The 0.5 release added the Criterion infrastructure but did not commit numeric baselines, and 0.9 keeps the same posture.
- **Resolution.** Baseline numbers are deferred to a 1.x release. Committing them now would pin a comparison reference taken on a developer machine, which is misleading. A proper baseline requires a stable CI runner or dedicated benchmark host. The benches themselves *are* in place, compile under `cargo bench --no-run`, and produce results when run locally.
- **Status.** Deferred to 1.x (does not block 1.0).
